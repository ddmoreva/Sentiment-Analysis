# Sentiment-Analysis
Автоматическое определение тональности (сентимент‑анализ) русскоязычных отзывов о товарах с применением глубоких нейросетевых моделей. 

Цель: создать систему для автоматического определения тональности отзывов (позитивный, негативный, нейтральный) с высокой точностью.

Задачи:

1. Собрать корпус отзывов о товарах из интернет‑магазинов (не менее 20 000 отзывов) или использовать открытые датасеты.

Был выбран датасет с отзывами на телефоны. Таблица с данными имеет более 320 тысяч строк, и это усложняет работу глубоких моделей обучения, именно поэтому было принято решение оставить только 25 тысяч строк. Ссылка на Kaggle: https://www.kaggle.com/datasets/theovall/phonereviews

2. Разметить данные по тональности (ручная или полуавтоматическая разметка).

Был выбран вариант полуавтоматической разметки, при которой модель проставляет метки POSITIVE, NEGATIVE, NEUTRAL автоматически, а нам лишь остается проверить и исправить часть данных (обычно сложные или сомнительные случаи). 

В основном, код прогоняет все отзывы через модель пачками (batch_size=32), чтобы ускорить процесс реализации. Первым делом проверяется наличие видеокарты с помощью torch.cuda.is_available: если GPU доступен, переменная device получает значение 0, что соответствует первому графическому процессору, в противном случае она устанавливается в -1, что означает использование центрального процессора. Затем создаётся объект *sentiment_pipeline* с указанием задачи "sentiment-analysis" и передачей имени модели "blanchefort/rubert-base-cased-sentiment" вместе с соответствующим токенизатором и параметром device, который определяет, где будут выполняться вычисления. 

Результатом работы пайплайна является список, содержащий словарь с ключами label и score, где label представляет предсказанную тональность (POSITIVE, NEGATIVE или NEUTRAL), а score – числовую уверенность модели в этом предсказании. Из датафрейма выбираются только три колонки: 'Review', 'sentiment' и 'confidence', и этот урезанный датафрейм сохраняется в CSV-файл с именем labeled_phone_reviews.csv. Важно отметить, что в данном проекте не рассматривался столбец с 'Rating', но можно рассмотреть в перспективе исследовать с сентимент-анализом также точность определения тональности с оценкой пользователя.

<u>Для оценивания ложного срабатывания были выбраны 100 строк и проверены на соотстветствие тональности, оказалось, что 8 из 100 были неверно определены. Можно утверждать, что если применить аппроксимацию, в итоге у 25 000 строк тональность будет определена некорректно у 2 000 строк. Это значительно влияет на обучение моделей, ведь, грубо говоря, они обучаются на некорректных данных.</u>

3. Провести предобработку текста: нормализацию, токенизацию, удаление шума.

Весь процесс предобработки текстовых данных начинается с загрузки датасета, содержащего исходные отзывы в колонке Review. Первым делом применяется функция нормализации, которая приводит каждый отзыв к нижнему регистру и удаляет все символы, кроме букв русского алфавита и пробелов, чтобы избавиться от посторонней пунктуации и латиницы; затем удаляются лишние пробелы. На следующем шаге происходит удаление шума: из текста вырезаются ссылки, адреса электронной почты и отдельно стоящие цифры, так как эти элементы не несут смысловой нагрузки для определения тональности. После очистки текст разбивается на отдельные слова с помощью простой токенизации по пробелам, что даёт список токенов, который сохраняется в колонке tokens. Далее выполняется лемматизация, которая является ключевым этапом для русского языка из‑за его богатой морфологии; для этого используется библиотека pymystem3, которая приводит каждое слово к его нормальной словарной форме: существительные к именительному падежу единственного числа, глаголы к инфинитиву и так далее. Чтобы ускорить обработку большого объёма данных, лемматизация применяется пакетами по 1000 отзывов, что позволяет сократить время выполнения. Результат лемматизации сохраняется в колонке lemmas. Затем из полученных лемм удаляются стоп-слова, то есть наиболее частотные и смыслово нейтральные единицы, такие как предлоги, союзы, частицы и местоимения; для этого используется стандартный список стоп-слов из библиотеки NLTK, адаптированный для русского языка. Удаление стоп-слов необходимо для классических моделей машинного обучения, таких как логистическая регрессия и наивный байесовский классификатор, чтобы уменьшить размерность признакового пространства и убрать шум, не влияющий на тональность.

4. Обучить модели сентимент‑анализа: классические ML‑модели (Logistic Regression, Naive Bayes); глубокие модели (LSTM, Transformer на базе ruBERT).

Сначала были обучены классические модели машинного обучения: логистическая регрессия и наивный байесовский классификатор, для которых тексты преобразовывались в числовые признаки с помощью TF-IDF векторизации с использованием униграмм и биграмм. Эти модели показали стабильные результаты, причём логистическая регрессия оказалась лучшей с точностью около 0.65. Затем мы обучили две глубокие модели: LSTM и Transformer на базе ruBERT. LSTM представляла собой рекуррентную нейронную сеть с двумя двунаправленными слоями и механизмами регуляризации для борьбы с переобучением. Модель ruBERT использовала предобученную русскоязычную версию BERT, которая была дообучена на датасете, однако из-за ограниченного объёма данных она не смогла превзойти логистическую регрессию. Хочется отметить, что ранее была разработана другая схема разделения токенов, и с ней ruBERT справлялась лучше - точность равнялась 0.74. После обучения всех моделей было проведено их сравнение по метрикам accuracy и F1-macro с построением матриц ошибок и графиков, что позволило визуализировать, какие классы каждая модель путает чаще всего.

5. Сравнить эффективность моделей по метрикам: accuracy, F1‑score, confusion matrix.

Итоговые результаты показали, что логистическая регрессия стала лучшей моделью с точностью 0.65, что объясняется сбалансированностью простоты и эффективности на датасете, тогда как глубокие модели не смогли полностью раскрыть свой потенциал из-за ограниченного объёма данных и дисбаланса классов. Для достижения более высокой точности необходимо увеличить размер выборки до 50-100 тысяч размеченных отзывов, улучшить качество разметки за счёт дополнительной ручной проверки и применить методы борьбы с дисбалансом классов, такие как взвешивание.

6. Исследовать влияние контекста и сленга на точность анализа.

Обнаружилось, что на 0.02 доли сленг влияет на точность определения тональности, поэтому для повышения точности можно также удалять сленг из токенов.

7. Реализовать API для интеграции системы в сторонние сервисы.

Для практического применения системы мы реализовали API с использованием FastAPI, создав эндпоинт для анализа тональности новых отзывов, который принимает текст и возвращает предсказанный класс и уверенность модели. API может быть запущен локально или развёрнут на облачных платформах для интеграции в сторонние сервисы.

Ожидаемый результат: модель с accuracy не менее 0,9, код проекта с документацией, отчёт с сравнением методов и рекомендациями по улучшению.

К сожалению, ожидаемого результата в цифру 0.9 тяжело добиться на таких данных, но в перспективу есть важные аспекты, на которые можно обратить внимание.
